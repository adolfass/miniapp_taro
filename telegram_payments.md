🤖 Инструкция для агента (Qwen code) по интеграции Stars
На основе этой документации и вашей текущей архитектуры (Node.js, Express, SQLite), вот четкий план действий.
1. Подготовка (Исследование)
Изучить методы Bot API: Внимательно прочитать разделы sendInvoice, answerPreCheckoutQuery и объект successful_payment в официальной документации Bot API - https://core.telegram.org/bots/api#payments 
Понять текущий код: Найти, где сейчас обрабатывается команда /start и инициализация бота (скорее всего, в server.js или отдельном файле bot.js). Интегрировать новые обработчики рядом.
2. Реализация процесса оплаты (Бэкенд)
Добавить обработчик команды для оплаты (например, по нажатию кнопки "Заказать консультацию за X Stars" во фронтенде):
Фронтенд отправляет запрос на бэкенд (POST /api/payment/create с userId и tarologistId).
Бэкенд создает уникальный payload для счета (например, JSON { userId: 123, tarologistId: 5 }).
Вызывается метод bot.telegram.sendInvoice:
chat_id: userId (кому выставляем счет).
title: "Консультация таролога".
description: "Персональная консультация с [Имя таролога]".
payload: уникальная строка (зашифрованный JSON с данными).
provider_token: "" (пустая строка!).
currency: "XTR".
prices: [{ label: "Консультация", amount: СТОИМОСТЬ_В_ЗВЕЗДАХ }].
Бот отправляет счет пользователю в личку. Сохраняем связку payload с userId и tarologistId в отдельной таблице pending_payments, чтобы потом понять, какой товар оплачен.
Обработчик pre_checkout_query:
Подписаться на событие on("pre_checkout_query").
Внутри получить query с данными счета.
Проверить, что товар еще доступен (например, таролог не удален, количество консультаций не исчерпано).
Если всё ок, вызвать answerPreCheckoutQuery(query.id, true).
Если ошибка, вызвать answerPreCheckoutQuery(query.id, false, { error_message: "Понятная причина отказа" }).
Обработчик successful_payment:
Подписаться на событие on("message") и проверять наличие поля successful_payment.
Получить payload из successful_payment.invoice_payload.
Это критический момент: расшифровать payload, найти userId и tarologistId.
Доставить товар:
Создать запись в таблице chat_sessions (или аналогичной) для этой консультации. Статус — "ожидает начала" или сразу "активна".
Уведомить пользователя (например, бот пишет: "Оплата прошла! Скоро с вами свяжется таролог @tarolog_username").
Уведомить таролога (бот пишет тарологу: "Новая оплаченная консультация от пользователя @username. Свяжитесь с ним в чате.").
Сохранить telegram_payment_charge_id в таблицу transactions для возможности возврата.
3. Интеграция с фронтендом
Создать в мини-приложении кнопку "💫 Купить консультацию за [X] Stars".
При нажатии кнопка вызывает POST /api/payment/create на бэкенд.
Бэкенд возвращает ответ об успешной отправке счета. Пользователь переходит в личку с ботом и нажимает "Оплатить".
Вся дальнейшая логика оплаты и подтверждения происходит в чате с ботом (это стандартный UX Telegram). Приложение может периодически опрашивать бэкенд на предмет изменения статуса сессии, чтобы показать, что оплата прошла.
4. Возвраты и поддержка
Реализовать команду бота /support, которая просто дает контакты администратора (или ссылку на чат с ним).
Реализовать команду бота /terms, которая отправляет красивое сообщение с пользовательским соглашением (текст нужно будет подготовить).
Для возврата средств:
Создать эндпоинт в админ-панели (и, возможно, команду для админа в боте).
Вызывать метод bot.telegram.refundStarPayment(userId, telegram_payment_charge_id).
Обновить статус транзакции в БД на refunded.
5. Тестирование
Использовать тестовую среду Telegram (тестовая сеть TON). Как к ней подключиться, описано в документации - https://core.telegram.org/bots/payments-stars#testing-payments.  Это важно, чтобы не тратить реальные звезды при отладке.
Протестировать полный цикл: нажатие кнопки → получение счета → оплата → получение successful_payment → доставка товара.
Протестировать отказ на этапе pre_checkout_query (например, "Таролог временно недоступен").
Протестировать возврат средств через админ-панель.
6. Меры безопасности и compliance
Проверка подлинности: Всегда проверять initData с фронтенда, но финальная верификация платежа происходит в обработчике successful_payment. Данные из initData можно использовать для удобства, но не для критических проверок.
Идемпотентность: Убедиться, что на один и тот же invoice_payload товар не будет доставлен дважды, если по какой-то причине придет два successful_payment.
Логирование: Подробно логировать все шаги: создание инвойса, получение pre_checkout_query, ответ на него, получение successful_payment, доставку товара. Это поможет при отладке и разрешении споров.
Эта часть инструкции основана на официальной документации Telegram Bot API - https://core.telegram.org/bots/api#payments  и содержит технические детали, необходимые для реализации.
1. Общие принципы работы с Bot API
Авторизация: Каждый запрос к API должен содержать токен бота в URL: https://api.telegram.org/bot<токен>/ИМЯ_МЕТОДА.
Формат запросов и ответов: Все запросы должны быть HTTPS. Ответ всегда приходит в формате JSON и содержит поле ok. Если ok: true, то результат лежит в поле result. Если false, в поле description будет описание ошибки, а в error_code — её код.
Получение обновлений (Updates): Бот должен получать события о действиях пользователей. Есть два способа:
Long polling (getUpdates): Бот сам периодически запрашивает сервер на предмет новых событий.
Webhook: Telegram сам отправляет POST-запросы с данными о новых событиях на указанный вами URL-эндпоинт (например, https://ваш-домен.ru/webhook). Этот способ предпочтительнее для сервера, который всегда в сети.
2. Технические детали реализации платежей (объекты и методы)
В дополнение к описанным ранее шагам, агенту нужно работать с конкретными типами данных:
A. Создание счета (sendInvoice)
При вызове этого метода необходимо сформировать объект счета. Ключевые поля для вашего случая:
chat_id: Уникальный идентификатор пользователя (Int или String).
title: Название товара (String, до 32 символов).
description: Описание товара (String, до 255 символов).
payload: Важно! Строка, которая будет передана обратно боту во всех последующих событиях по этому счету. Именно сюда мы сохраняем наши данные (userId и tarologistId). Максимальная длина — 128 байт.
provider_token: Для Telegram Stars (цифровых товаров) это поле должно быть пустой строкой ("").
currency: Трехбуквенный код валюты. Для Stars это всегда "XTR".
prices: Массив объектов LabeledPrice. Каждый объект имеет поля label (название позиции) и amount (цена в минимальных единицах валюты). Для Stars amount — это просто целое число, равное цене в звездах (т.е. для счета в 50 звезд amount: 50).
start_parameter: Этот параметр управляет поведением счета при пересылке. В вашем случае (продажа консультаций в личке) лучше использовать уникальное значение, чтобы предотвратить оплату одного счета несколько раз (или использовать как вам нужно).
B. Обработка PreCheckoutQuery
Это событие приходит в виде объекта PreCheckoutQuery. Вот его ключевые поля, которые нужно обработать:
json
{
  "id": "уникальный_id_запроса",
  "from": { ... }, // Объект User, кто платит
  "currency": "XTR",
  "total_amount": 50, // Сумма в звездах
  "invoice_payload": "СТРОКА_ИЗ_ВАШЕГО_ПЛАТЕЖА", // Тот самый payload!
  ... // и другие поля
}
После получения этого события бот должен в течение 10 секунд ответить методом answerPreCheckoutQuery, передав pre_checkout_query_id и флаг ok (true — одобрить, false — отклонить). При отклонении можно передать понятное пользователю сообщение в поле error_message.
C. Обработка SuccessfulPayment
Это поле находится внутри объекта Message, которое бот получает как часть обычного апдейта message. Его структура:
json
{
  "message_id": ...,
  "from": { ... }, // Кто прислал сообщение (пользователь)
  "chat": { ... },
  "date": ...,
  "successful_payment": {
    "currency": "XTR",
    "total_amount": 50,
    "invoice_payload": "СТРОКА_ИЗ_ПЛЕЙЛОАДА",
    "telegram_payment_charge_id": "идентификатор_платежа_от_телеграм",
    ... // и другие поля
  }
}
Получив это сообщение, вы извлекаете invoice_payload, находите в своей базе userId и tarologistId и доставляете услугу.
3. Обработка возвратов
Для возврата средств используется метод refundStarPayment. Ему нужно передать:
user_id: ID пользователя, которому возвращаем.
telegram_payment_charge_id: Тот самый идентификатор платежа, который вы сохранили при успешной оплате.
4. Тестовая среда
Для отладки обязательно используйте тестовую среду Telegram. В ней у вас будет тестовый бот, тестовые пользователи и тестовые звезды, что позволит безопасно проверить все сценарии, не тратя реальные средства. Инструкция по подключению к тестовой среде есть в документации - https://core.telegram.org/bots/payments-stars#testing-payments 
5. Важные замечания по надежности
Идемпотентность и повторная доставка: Технически Telegram может иногда прислать один и тот же апдейт дважды. Поэтому, прежде чем доставлять товар по successful_payment, проверьте в своей базе, не был ли этот telegram_payment_charge_id уже обработан.
Обработка ошибок API: Всегда проверяйте ответы от Bot API. В случае ошибок сети или таймаутов используйте логику повторных попыток (retry) с экспоненциальной задержкой.
