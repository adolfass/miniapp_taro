# ADR-003: Управление жизненным циклом сессий консультаций

## Статус
Accepted

## Контекст
В системе обнаружена критическая проблема: сессии чата остаются в статусе `active = 1` бесконечно долго, что приводит к:
1. Ложному онлайн-статусу тарологов (isRealOnline() возвращает true из-за активных сессий)
2. Накоплению мусорных данных в БД
3. Некорректной статистике

## Решение

### 1. Автоматическое закрытие сессий по таймеру (25 минут)

**Техническая реализация:**
```javascript
// db.js - ChatSession.autoCloseOldSessions()
// Закрывает все сессии старше 25 минут:
// - active = 0
// - end_time = CURRENT_TIMESTAMP
// - completed = 1 (если не было досрочного закрытия)
// - rated = 0 (флаг для статистики - пользователь не оценил)
```

**Где вызывать:**
- При каждом запросе статуса таролога ( middleware / хук )
- При открытии чата клиентом
- Cron-задача раз в 5 минут (опционально)

### 2. Модальное окно оценки после сессии

**Flow:**
```
Сессия началась (оплата прошла)
    ↓
25 минут чата
    ↓
Автоматическое закрытие сессии
    ↓
Показ модального окна "Оцените консультацию"
    ↓
Пользователь ставит 1-5 звёзд
    ↓
Сохранение оценки в БД
```

**Если пользователь отказывается оценить:**
- Модаль закрывается по "×" или таймауту 30 секунд
- Сессия помечается `rated = 0` (без оценки)
- В статистике таролога: "Пользователь не поставил оценку"

### 3. Структура БД

**Таблица chat_sessions:**
```sql
ALTER TABLE chat_sessions ADD COLUMN rated BOOLEAN DEFAULT 0;
ALTER TABLE chat_sessions ADD COLUMN rating INTEGER; -- 1-5 звёзд
ALTER TABLE chat_sessions ADD COLUMN rating_comment TEXT; -- текстовый отзыв
ALTER TABLE chat_sessions ADD COLUMN rated_at DATETIME;
```

**Таблица tarologists (для статистики):**
- Уже есть: rating, total_ratings, sessions_completed
- Добавить: sessions_without_rating (количество сессий без оценки)

### 4. Обновление isRealOnline()

```javascript
isRealOnline(id) {
  // ... предыдущие проверки ...
  
  // 4. Проверка активной сессии чата (ТОЛЬКО не старше 25 минут!)
  const activeSession = db.prepare(`
    SELECT COUNT(*) as count 
    FROM chat_sessions 
    WHERE tarologist_id = ? 
      AND active = 1
      AND datetime(start_time, '+25 minutes') > datetime('now')
  `).get(id);
  
  if (activeSession.count > 0) {
    return true;
  }
  
  return false;
}
```

### 5. API Endpoints

**POST /api/session/:id/rate**
- Принимает: rating (1-5), comment (опционально)
- Обновляет сессию: rated = 1, rating, rating_comment, rated_at
- Обновляет статистику таролога

**GET /api/session/:id/status**
- Возвращает: is_active, time_left (секунд до закрытия), can_rate (можно ли оценить)

### 6. Frontend: Rating Modal

**Компонент:** `RatingModal`
**Появляется:**
- После автоматического закрытия сессии
- При попытке закрыть чат вручную ("Завершить досрочно")

**UI:**
- 5 звёзд для выбора
- Поле для комментария (опционально)
- Кнопка "Пропустить" (отказ от оценки)
- Таймаут 30 секунд → авто-закрытие

## Последствия

### Положительные:
- Корректный онлайн-статус тарологов
- Чистая БД без зависших сессий
- Сбор обратной связи от клиентов
- Прозрачная статистика для админки

### Негативные:
- Нужна миграция БД
- Дополнительная логика на фронтенде
- Обработка случаев, когда клиент закрыл приложение до оценки

## Альтернативы (отклонённые)

### Альтернатива A: Ручное закрытие сессий тарологом
- Таролог должен нажать "Завершить консультацию"
- **Отклонено:** Не гарантирует закрытие, таролог может забыть

### Альтернатива B: Только WebSocket для отслеживания активности
- Закрывать сессию при отключении WebSocket
- **Отклонено:** Ненадёжно, пользователь может переподключиться

### Альтернатива C: Без оценки
- Просто закрывать сессии по таймеру
- **Отклонено:** Теряем ценную обратную связь

## Реализация

### Phase 1: Backend (db.js, server.js)
1. Миграция БД (новые колонки)
2. Метод autoCloseOldSessions()
3. Обновление isRealOnline()
4. API для оценки

### Phase 2: Frontend (chat.js, rating-modal.js)
1. Компонент RatingModal
2. Интеграция с чатом
3. Обработка таймаута сессии

### Phase 3: Cleanup
1. Очистить существующие зависшие сессии
2. Протестировать полный flow
3. Обновить документацию

## Миграция данных

```sql
-- Очистить зависшие сессии старше 25 минут
UPDATE chat_sessions 
SET active = 0, 
    completed = 1, 
    end_time = datetime(start_time, '+25 minutes'),
    rated = 0
WHERE active = 1 
  AND datetime(start_time, '+25 minutes') < datetime('now');
```
